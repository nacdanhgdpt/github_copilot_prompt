# Bảng Thuật ngữ Chuẩn - Vietnamese Translation Reference

## Mục đích
Tài liệu này cung cấp bảng thuật ngữ chuẩn để đảm bảo tính nhất quán trong quá trình dịch các file của dự án sang tiếng Việt.

## Nguyên tắc Dịch thuật

### 1. Giữ nguyên YAML Frontmatter
- Tất cả YAML frontmatter phải được giữ nguyên 100%
- Ví dụ: `---\nmode: agent\n---` không được thay đổi

### 2. Giữ nguyên Code Blocks
- Tất cả code trong code blocks phải được giữ nguyên
- Chỉ dịch các comment giải thích nếu cần thiết

### 3. Giữ nguyên Commands và Technical Syntax
- Tất cả lệnh command-line phải được giữ nguyên
- Ví dụ: `git commit -m "message"` không được thay đổi

### 4. Giữ nguyên URLs và File Paths
- Tất cả URLs, đường dẫn file, và tham chiếu kỹ thuật phải được giữ nguyên

## Hướng dẫn Sử dụng

### Khi nào giữ nguyên thuật ngữ tiếng Anh?
1. Thuật ngữ đã trở thành chuẩn quốc tế (API, Git, Docker)
2. Thuật ngữ không có bản dịch tiếng Việt phổ biến
3. Dịch sang tiếng Việt sẽ làm mất đi ý nghĩa kỹ thuật
4. Thuật ngữ xuất hiện trong code hoặc command

### Khi nào dịch sang tiếng Việt?
1. Các từ mô tả chung (requirements, design, task)
2. Các hướng dẫn và giải thích
3. Các user story và acceptance criteria
4. Nội dung văn bản không phải kỹ thuật

### Khi nào kết hợp cả hai?
1. Lần đầu tiên xuất hiện: "Pull Request (PR - Yêu cầu kéo)"
2. Các lần sau: "Pull Request (PR)" hoặc chỉ "Pull Request"
3. Tùy thuộc vào ngữ cảnh và độ phổ biến của thuật ngữ

## Ví dụ Áp dụng

### Ví dụ 1: Dịch hướng dẫn
```markdown
# Tiếng Anh
You must create a Pull Request for code review.

# Tiếng Việt
Bạn phải tạo Pull Request để xem xét mã nguồn.
```

### Ví dụ 2: Giữ nguyên code
```markdown
# Tiếng Anh
Run the following command:
```bash
git commit -m "Add feature"
```

# Tiếng Việt
Chạy lệnh sau:
```bash
git commit -m "Add feature"
```
```

### Ví dụ 3: Kết hợp
```markdown
# Tiếng Anh
The agent will process your prompt and generate code.

# Tiếng Việt
Agent sẽ xử lý prompt của bạn và tạo mã nguồn.
```

## Lưu ý Quan trọng

1. **Tính nhất quán**: Luôn sử dụng cùng một cách dịch cho cùng một thuật ngữ trong toàn bộ dự án
2. **Ngữ cảnh**: Xem xét ngữ cảnh trước khi quyết định dịch hay giữ nguyên
3. **Độ phổ biến**: Ưu tiên giữ nguyên các thuật ngữ đã phổ biến trong cộng đồng lập trình viên Việt Nam
4. **Tính tự nhiên**: Đảm bảo bản dịch nghe tự nhiên và dễ hiểu cho người Việt
5. **Chính xác kỹ thuật**: Không được làm mất đi ý nghĩa kỹ thuật của thuật ngữ gốc
